# SDK 化前主体项目优化计划

> 本文件用于给执行 agent 按项修改。  
> 当前阶段仍然先优化主体项目，不恢复 C 风格导出层，不做打包发布。  
> 执行 agent 每完成一项，必须把对应 `[ ]` 改成 `[x]`，并在文末记录实际验证命令和结果。完成后再交回审查。

## 一、总体方向

当前项目已经完成四步校准主链路：

```text
Closed -> Fist -> Spread -> Crosstalk -> isReady()
```

后续若要打包成 SDK，主体项目需要先解决三个问题：

1. 调用状态必须清晰，错误阶段不能拖到最后才失败。
2. 失败原因必须可诊断，不能只靠 `bool` 猜原因。
3. 校准结果必须可获取，客户需要知道无效通道、串扰质量和失败原因。

本轮只做最短路径主体优化，不做兼容层，不恢复 `matrix_hand_sdk.h/.cpp`。

## 二、任务 1：补齐主体算法状态与结果类型

### 问题

`HandAngleAlgorithm` 当前大量接口只返回 `bool`：

- `setRuntimeConfig()`
- `pushCalibrationFrame()`
- `finishCalibration()`
- `processFrame()`

这对 demo 足够，但 SDK 化前不够。调用方无法区分：

- 配置非法
- 阶段顺序错误
- 未开始采样
- 空采样
- Crosstalk 拟合失败
- 未完成四步校准
- 输入指针为空

### 解决方案

- [ ] 在 `hand_algorithm.h` 新增 C++ 主体层状态枚举，不使用 `MATRIX_HAND_*` 旧 C API 名称。

建议：

```cpp
enum class AlgorithmStatus : int32_t {
    Ok = 0,
    NullInput,
    InvalidConfig,
    BadState,
    BadCalibrationStage,
    CalibrationNotActive,
    CalibrationEmpty,
    NotReady,
    CrosstalkFitFailed,
};
```

- [ ] 新增校准结果结构，用于承载每次 `finishCalibration()` 的结果。

建议：

```cpp
struct CalibrationResult {
    CalibrationStage stage = CalibrationStage::Closed;
    AlgorithmStatus status = AlgorithmStatus::Ok;
    std::size_t frameCount = 0;
    std::vector<int> invalidChannels;
    std::vector<int> xtalkUnstableChannels;
    std::size_t xtalkValidTargetChannelCount = 0;
};
```

- [ ] 保留现有 `bool` 接口也可以，但必须新增可诊断接口，避免破坏 main.cpp 的最短验证链路。

建议新增：

```cpp
AlgorithmStatus setRuntimeConfigEx(const RuntimeConfig& runtimeConfig);
AlgorithmStatus beginCalibrationEx(CalibrationStage stage);
AlgorithmStatus pushCalibrationFrameEx(const int16_t adValues[kChannelCount]);
AlgorithmStatus finishCalibrationEx(CalibrationResult& result);
AlgorithmStatus processFrameEx(const int16_t adValues[kChannelCount], HandAngleOutput& outputValue);
```

旧 `bool` 接口可以内部调用新接口并转换为 `status == AlgorithmStatus::Ok`，不要保留两套业务逻辑。

### 验收标准

- [ ] `hand_algorithm.h` 中存在 `AlgorithmStatus` 和 `CalibrationResult`。
- [ ] 新增 `*Ex` 接口，旧 `bool` 接口只做薄转换，不重复业务判断。
- [ ] 所有失败路径能给出明确 `AlgorithmStatus`。
- [ ] 不出现 `MATRIX_HAND_`、`matrix_hand_`、`MatrixHandHandle`。

## 三、任务 2：begin 阶段前置校验前移

### 问题

当前 `beginCalibration(stage)` 会直接开始任意阶段，阶段顺序错误通常到 `finishCalibration()` 才失败。SDK 调用方会更难定位错误。

### 解决方案

- [ ] `beginCalibrationEx(stage)` 必须在开始采样前校验状态和阶段顺序。

规则：

```text
未完成任何阶段 -> 只允许 Closed
已完成 Closed -> 只允许 Fist
已完成 Closed/Fist -> 只允许 Spread
已完成 Closed/Fist/Spread -> 只允许 Crosstalk
四步完成 -> 不允许继续 begin，除非先 reset()
已有采样进行中 -> 返回 BadState
非法枚举值 -> 返回 BadCalibrationStage
```

- [ ] 旧 `beginCalibration(stage)` 改为调用 `beginCalibrationEx(stage)`，忽略返回值或保持 void，但不能绕过校验逻辑。

### 验收标准

- [ ] 直接 `beginCalibrationEx(Fist)` 返回 `BadCalibrationStage`，且不进入采样状态。
- [ ] 直接 `beginCalibrationEx(Crosstalk)` 返回 `BadCalibrationStage`。
- [ ] 正常四步顺序全部返回 `Ok`。
- [ ] 四步完成后再次 begin 返回 `BadState` 或 `BadCalibrationStage`，但不能开启新采样。

## 四、任务 3：finish 返回完整校准结果

### 问题

当前 `finishCalibration()` 只返回 `bool`。Fist/Spread 的无效通道、Crosstalk 的异常通道、有效拟合通道数没有统一结果对象。

### 解决方案

- [ ] `finishCalibrationEx(CalibrationResult& result)` 填充当前阶段、帧数和状态。
- [ ] Closed 阶段成功时，`invalidChannels` 为空。
- [ ] Fist 阶段成功时，检查弯曲相关通道、CH16、门控通道，把无效通道写入 `invalidChannels`。
- [ ] Spread 阶段成功时，检查 CH4/8/12/16，把无效通道写入 `invalidChannels`。
- [ ] Crosstalk 阶段成功时，写入 `xtalkValidTargetChannelCount` 和 `xtalkUnstableChannels`。
- [ ] Crosstalk 拟合失败时，返回 `CrosstalkFitFailed`，`isReady()` 仍为 `false`。

### 验收标准

- [ ] Closed 成功：`status == Ok`，`frameCount > 0`，`invalidChannels.empty()`。
- [ ] Fist 中某弯曲通道变化小于阈值：该通道出现在 `invalidChannels`，通道编号为 1-based。
- [ ] Spread 中某展开通道变化小于阈值：该通道出现在 `invalidChannels`，通道编号为 1-based。
- [ ] Crosstalk 成功：`xtalkValidTargetChannelCount > 0`。
- [ ] Crosstalk 失败：`status == CrosstalkFitFailed`，`isReady() == false`。

## 五、任务 4：processFrame 和输入校验语义明确

### 问题

当前 `processFrame()` 对空指针和未 ready 都返回 `false`。后续 SDK 化时，调用方需要知道具体原因。

### 解决方案

- [ ] `processFrameEx()` 对 `adValues == nullptr` 返回 `NullInput`，并清零输出。
- [ ] `processFrameEx()` 在未 ready 时返回 `NotReady`，并清零输出。
- [ ] `processFrameEx()` 成功时返回 `Ok`。
- [ ] 明确 AD 输入范围策略：本轮不做 clamp，不静默修正。若发现任一 AD 不在 `[kAdMinValue, kAdMaxValue]`，建议返回 `InvalidInput`；如果不新增该状态，则先在文档说明 AD 范围由调用方保证。

如采用输入范围校验，则补充状态：

```cpp
InvalidInput,
```

### 验收标准

- [ ] 空指针输入：返回 `NullInput`，输出清零。
- [ ] 未 ready 输入：返回 `NotReady`，输出清零。
- [ ] 四步 ready 后输入：返回 `Ok`。
- [ ] AD 范围策略在代码和文档中保持一致。
